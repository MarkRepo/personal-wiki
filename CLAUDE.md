# Personal Wiki Schema (CLAUDE.md)

> 本文件定义 LLM 维护此知识库的行为规范。每次操作前必须读取。

## 目录结构

```
personal-wiki/
├── raw/          原始文档，只读，LLM 不可修改
│   └── assets/       本地化图片存放目录
├── wiki/         LLM 维护的知识库
│   ├── entities/     具体事物：人物、公司、产品、地点（书籍不在此，见 synthesis/）
│   ├── concepts/     抽象概念：理论、方法、思想、术语
│   ├── events/       事件：历史事件、新闻、里程碑
│   ├── relations/    对比与关联分析（如 A_vs_B.md）
│   ├── synthesis/    综合视图、领域概述、读书笔记汇总
│   ├── macro/        宏观主题：趋势、时代背景、长周期规律
│   ├── index.md      全局索引，每次 ingest 后更新
│   └── log.md        操作日志，只追加
├── inbox/        新文档暂存区，处理后移至 raw/
└── scripts/      自动化脚本
```

## 页面格式规范

> **所有页面的统一目标**：读者看完此页，无需查阅原始资料，也能精准、完整地理解该页面的核心内容。每种页面都有对应的自查标准，写完必须自查通过再提交。

### 实体页 (entities/NAME.md)
适用于：人物、公司、产品、地点、电影等具体事物（**书籍不建实体页，只建读书笔记页**）

```markdown
---
name: 
type: person | company | product | place | other
tags: []
related: []
last_updated: YYYY-MM-DDTHH:MM:SS
source_count: 0
---

## 概述
<!-- 是什么，一句话定义 -->

## 核心内容
<!-- 最重要的事实、观点、数据 -->

## 背景与历史
<!-- 来龙去脉 -->

## 影响与评价
<!-- 对世界/领域的影响，争议观点并列记录 -->

## 关联页面
<!-- [[concepts/XXX]] [[events/XXX]] -->
```

写完后自查：

| 维度 | 标准 |
|------|------|
| 特 | 内容专属于该实体，换一个相似实体就不成立 |
| 全 | 覆盖实体的全部主要方面，无明显偏科 |
| 实 | 有具体数据、案例或事件，不停留在摘要层 |

### 概念页 (concepts/NAME.md)
适用于：理论、方法论、思想框架、学科术语

```markdown
---
name: 
domain: []   # 所属领域，如 [物理, 信息论]
tags: []
related: []
last_updated: YYYY-MM-DDTHH:MM:SS
---

## 定义
<!-- 精确定义，一段话 -->

## 核心原理
<!-- 关键机制和逻辑 -->

## 应用场景
<!-- 在哪里用，怎么用 -->

## 局限性
<!-- 适用边界，常见误解 -->

## 关联页面
```

写完后自查：

| 维度 | 标准 |
|------|------|
| 清 | 定义精确，能区分于相似概念，不循环定义 |
| 透 | 机制说透，不只命名，说清内部逻辑（为什么/如何运作） |
| 用 | 读完能用来分析真实问题，有具体应用路径 |
| 联 | 与相关概念的来源、依赖、对比关系清楚 |

### 综合页 (synthesis/NAME.md)
适用于：领域概述、主题研究报告、课程笔记等

**两种子类型：**
- **读书笔记**（`type: book-note`）：单本书籍的忠实提炼，见下节
- **主题综合页**（`type: thematic`）：跨多个输入源的知识重组，输入来自 wiki 内已有页面

主题综合页的内容来源只能是 wiki 内已有页面（读书笔记、概念页、实体页等），不引入外部资料或模型训练知识。

```markdown
---
title: 
type: thematic
scope: []   # 覆盖的实体/概念
tags: []
last_updated: YYYY-MM-DDTHH:MM:SS
sources:    # 本页直接依赖的 wiki 页面；源页面更新后需审查本页
  - synthesis/XXX读书笔记
  - concepts/XXX
---

## 核心结论
## 详细分析
## 知识地图
<!-- 本主题下的关键页面列表 -->
## 关联页面
```

**主题综合页的新建时机**

满足以下条件时建议新建：

- **至少两个有实质重叠的输入源**：各源从不同角度切入同一问题，跨源比较能产生单本读书笔记没有的洞察；若只有一个源，读书笔记已够用
- **主题足够聚焦**：有共同的核心问题，而非宽泛的学科标签（"经济学"不够，"债务周期的传导机制"才够）

自然触发点（由 LLM 在 ingest 或 query 时主动提示，由用户决定是否执行）：
- Ingest 新书后发现与已有 1-2 本书在同一主题上有实质收敛
- Query 回答需同时引用 3 本以上的书，说明主题已有足够积累
- Lint 报告某概念被多个源提及但缺乏统一综合视角

不适合建的情况：主题过宽泛、各源角度完全平行没有张力（综合页会退化成目录）。

### 读书笔记页 (synthesis/NAME读书笔记.md)
适用于：书籍、长篇报告等独立完整的作品（**书籍的唯一 wiki 入口，不另建实体页**）

> **目标**：读者看完此页，无需读原书，也能精准、完整、无错漏地 get 全书核心。

结构、章节名、内容取舍完全自由，服从书的内在逻辑，不套模板。

Frontmatter 保留：

```markdown
---
title: 
type: book-note
tags: []
last_updated: YYYY-MM-DDTHH:MM:SS
ingest_model: claude-sonnet-4.6
---
```

写完后，以"从未读过原书的读者"视角逐项自查，全部通过再提交：

| # | 维度 | 检验标准 |
|---|------|----------|
| 准 | 事实准确 | 无人物/情节/论点/数据错误，不曲解作者原意，不张冠李戴，不主观脑补 |
| 核 | 抓住核心 | 紧扣核心主题、核心观点、主线逻辑，不纠结细枝末节，不跑偏无关内容 |
| 精 | 极简浓缩 | 不照搬原文、不堆砌废话，用最少文字讲清关键，读最少字掌握核心 |
| 逻 | 脉络清晰 | 不是零散碎片，能理清书的论证/叙事顺序，逻辑自洽，各部分关系清楚 |
| 专 | 内容专属 | 不套通用模板，不是所有书都能用的空话，内容是这本书专属的核心总结 |
| 全 | 无遗漏 | 关键结论、核心论据、重要转折无遗漏；次要细节可省，主干不丢 |

在正文叙述中，凡讲到某个衍生 concept/实体页面对应的内容时，在该处内联引用（`[[concepts/XXX]]`、`[[entities/XXX]]`），不在末尾单独列举列表。**无需强制覆盖所有衍生页面**——若某页面与读书笔记核心叙事无自然交汇点，保留在末尾关联页面中即可。叙事结构服从书的内在逻辑，引用跟着叙事走，不能为了覆盖引用而新建章节。

## 数据源处理

特定数据源的获取和转换流程通过 **Skills** 按需加载，不在本文件中展开：

| 数据源 | Skill | 触发方式 |
|--------|-------|----------|
| YouTube 视频 | `/youtube <url>` | 手动或遇到 YouTube URL 自动触发 |
| EPUB 电子书 | `/epub <path>` | 手动或遇到 .epub 文件自动触发 |

Skill 文件位于 `~/.claude/skills/`（全局，跨知识库共用），包含完整的工具链、命令和踩坑经验。

## 大文件 Ingest 策略（书籍/长文档）

转换后的 markdown 可能很大，需在读取前判断策略。**Read 工具单次上限约 80 行（~2600 tokens）**，但多次 Read 的结果可以全部积累在同一个 context 里，最后一次性综合。

**估算公式**：`token 估算 ≈ 文件字节数 ÷ 4`（中英混合 markdown 经验值）

```bash
wc -c raw/书名_converted.md   # 查文件字节数，除以 4 得 token 估算
```

**架构原则**：主 Agent 只做 wiki 操作（读 index、读已有页面、写输出），不读入源文档正文。源文档交给子 Agent 读取并返回摘要。

### 策略一：主 Agent 直接读（源文档估算 < 50k tokens）

适用于文章、字幕、短报告等短文档：

1. `grep -n "^#" 文件` 确认结构（可选）
2. 并行发出所有 Read 调用，将全文载入 context
3. 综合写 wiki 页面

### 策略二：子 Agent 读取（源文档估算 ≥ 50k tokens）

适用于书籍、长报告等长文档：

1. `grep -n "^#" 文件` 扫描语义边界，按章节分组
2. **每组书籍内容 ≤ 150k tokens（约 600k 字节）**；保证整章不拆断，在此前提下尽量少分组
3. 无标题结构时（如纯文本），改用日期/序号等显式边界分组
4. 每组一个子 Agent，**组内并行发出所有 Read**，读完后**只输出本组摘要**（3-5k tokens），不写 wiki
5. 主 Agent 读取所有组的摘要，再读相关 wiki 页面，最后统一写读书笔记


## Ingest 操作规范

当收到新文档（文章、笔记、书摘、网页等）时，执行：

1. **读取 wiki/index.md**，了解当前已有哪些页面
2. **分析文档**，识别涉及的实体、概念、事件
3. **筛选关键图片**（如文档含图片引用）：
   - 扫描源 markdown 中的所有 `![](...)` 引用
   - **查看每张图片**，分为三类：
     - **关键图表**：数据表格、流程图、模型图、架构图、趋势曲线等——传递核心信息，文字难以替代
     - **辅助插图**：示意图、照片等——有参考价值但非必需
     - **装饰性图片**：封面、分隔符、出版信息、logo——跳过不引用
   - 记录关键图表的路径和所属上下文（在哪个章节/段落旁），后续步骤中写入 wiki 页面
4. **读取相关已有页面**（最多 3-5 个）
5. **更新或新建页面**：
   - 新内容：按模板新建页面，归入对应子目录
   - **已有页面必须实质更新，不能只加链接**：
     - 新文档提供了新的事实、数据、观点、指标 → 写入对应章节（核心内容/应用场景/确认指标等），标注来源和日期
     - 新文档提供了新的关联关系但无增量内容 → 仅更新关联页面章节
     - 判断标准：如果读者只看这个已有页面，能否感知到新文档带来的信息增量？如果不能，说明更新不到位
   - **嵌入关键图片**：在相关段落处插入 `![简要描述](raw/assets/书名/ImageXXX.jpg)`，并附一行文字说明图片内容
6. **更新 wiki/index.md**：新页面加入索引，更新摘要和时间
7. **追加 wiki/log.md**：`## [YYYY-MM-DD] ingest | 文档标题`
8. **移动原始文档**：将 inbox 中的原始文档（epub/网页/html 等）以及 pandoc 转换后的 `.md` 文件全部移至 `raw/`，确保 inbox 清空
9. **更新 qmd 索引**：运行 `qmd embed` 为新增/修改的页面生成向量索引
10. **Ingest 质量评分**（书籍/长篇内容时执行，短文章可跳过）：
    评分是**整体过程**，分两层，顺序执行：

    **第一层（子过程）：各衍生页面自查**
    逐一按该页面类型的自查标准检查（概念页：清/透/用/联；实体页：特/全/实）。
    子页面质量直接影响读书笔记的"全"维度——子页面不达标须先修正再评分。

    **第二层（主评分）：读书笔记六维评分**
    评分对象是读书笔记正文 **+ 所有衍生子页面的内容**，视为一个整体。
    各维各打 1-10 分，追加到 `wiki/ingest_scores.md`：
    - **准**：无事实错误，不曲解作者原意，不脑补（含子页面）
    - **核**：紧扣核心主题与主线逻辑，不跑偏
    - **精**：极简浓缩，无废话，最少字讲清关键
    - **逻**：脉络清晰，论证/叙事顺序可循，逻辑自洽
    - **专**：内容是本书专属，不套通用模板空话
    - **全**：核心结论/论据/转折无遗漏，主干完整（子页面贡献的内容纳入"全"的考量）

    `wiki/ingest_scores.md` 格式（追加一行）：
    ```
    | YYYY-MM-DD | 模型名 | 书名 | 准 | 核 | 精 | 逻 | 专 | 全 | 总分 | 备注 |
    ```
    评分后直接告知用户六项得分及简短理由。

11. **综合页影响评估**（re-ingest 或有新页面产生时执行）：
    检查本次 ingest 是否影响已有主题综合页，分两类：
    - **内容变更**：本次修改的页面出现在某综合页的 `sources` 列表中 → 标记该综合页需审查
    - **范围扩展**：本次新建的页面在主题上与某综合页高度相关，但尚未被纳入 → 建议将新页面加入该综合页的 `sources` 并更新内容

    输出格式：
    ```
    受影响的综合页：
    - synthesis/XXX（原因：sources 中的 YYY 已更新 / 新建页面 ZZZ 可能属于此主题）
    ```
    由用户决定是否触发综合页更新，不自动修改。

12. **输出**：列出所有新建/修改的文件，标注引用了哪些图片

**注意：**
- **所有写入 wiki 的内容必须来自 ingest 的原始文档**，不得使用模型训练知识或 web 搜索结果填充。书中出现的概念、数据、案例，必须能在原文中找到对应依据；原文未提及的背景知识一律不写入
- 观点矛盾时，并列记录并注明来源，不擅自判断
- 数据信息标注来源和日期，避免过时数据覆盖新数据
- 一次 ingest 通常涉及 2-8 个页面
- 图片引用本地路径（`raw/assets/书名/filename.jpg`），避免依赖外部 URL
- 图片数量多时（>20张），优先筛选关键图表，辅助插图酌情引用，不必全部查看
- **ingest 完成后 inbox 中对应文档必须移走，不得残留**

## Query 操作规范

1. 读取 wiki/index.md 找相关页面
2. 读取相关页面内容
3. 基于 wiki 内容回答，标注引用（[[页面名]]）
4. **有价值的分析结论默认建议写回 wiki**（对比、洞察、发现的关联等），不应让知识停留在聊天记录里；由用户决定是否执行
5. 追加 log.md：`## [YYYY-MM-DD] query | 问题摘要`

## Lint 操作规范（每月或按需执行）

检查并报告：
- 孤立页面（无 inbound links）
- 数据超过 90 天未更新
- 页面间存在明显矛盾的观点
- 重要概念被提及但无独立页面
- 建议新增的对比页或综合页
- **主题综合页过时检测**：`sources` 中任意页面的 `last_updated` 晚于综合页的 `last_updated`，报告为"内容可能过时"；`sources` 中未包含但主题高度相关的新页面，报告为"可能有遗漏"

追加 log.md：`## [YYYY-MM-DD] lint | 发现 N 个问题`

> log.md 格式说明：每条记录以 `## [YYYY-MM-DD]` 开头，便于用 `grep "^## \[" log.md` 解析历史。

## 搜索策略

- **小规模（< 100 页）**：直接读 index.md 导航，无需额外工具
- **中大规模（>= 100 页）**：考虑引入 [qmd](https://github.com/tobi/qmd)，提供本地 BM25/向量混合搜索，支持 CLI 和 MCP 两种接入方式

## 命名规范

- 人物：中文姓名或英文全名，如 `爱因斯坦.md`、`Alan_Turing.md`
- 公司/产品：常用名，如 `OpenAI.md`、`iPhone.md`
- 概念：准确术语，如 `熵.md`、`贝叶斯推断.md`
- 事件：简明描述，如 `2008年金融危机.md`、`阿波罗11号登月.md`
- 对比：`A_vs_B.md`
- 综合/宏观：中文名，如 `人工智能概览.md`

## 语言规范

- 默认用中文写 wiki 页面
- 专有名词、术语首次出现时附英文原文，如 `熵（Entropy）`
- 数字和数据标注单位和来源
