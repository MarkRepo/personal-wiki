# Personal Wiki Schema (CLAUDE.md)

> 本文件定义 LLM 维护此知识库的行为规范。每次操作前必须读取。

## 目录结构

```
personal-wiki/
├── raw/          原始文档，只读，LLM 不可修改
│   └── assets/       本地化图片存放目录
├── wiki/         LLM 维护的知识库
│   ├── entities/     具体事物：人物、公司、产品、地点、书籍
│   ├── concepts/     抽象概念：理论、方法、思想、术语
│   ├── events/       事件：历史事件、新闻、里程碑
│   ├── relations/    对比与关联分析（如 A_vs_B.md）
│   ├── synthesis/    综合视图、领域概述、读书笔记汇总
│   ├── macro/        宏观主题：趋势、时代背景、长周期规律
│   ├── index.md      全局索引，每次 ingest 后更新
│   └── log.md        操作日志，只追加
├── inbox/        新文档暂存区，处理后移至 raw/
└── scripts/      自动化脚本
```

### 读书笔记页 (synthesis/NAME读书笔记.md)
适用于：书籍、长篇报告等独立完整的作品

> **核心目标**：读者看完此页，无需读原书，也能精准、完整、无错漏地 get 全书核心。

**这是一份压缩的书，不是目录。** 结构服从书的内在逻辑，不按模板填空。读完全书后，自由决定如何组织内容。

#### Frontmatter（保留）

```markdown
---
title: 
type: book-note
tags: []
last_updated: YYYY-MM-DD
ingest_model: claude-sonnet-4.6
---
```

#### 必须回答的问题（不规定 section 名，自行决定如何组织）

1. **这本书在解决什么问题？** 作者的出发点，一句话定位
2. **作者的核心答案是什么？** 最重要的结论，不超过 5 条
3. **作者怎么论证的？** 完整的因果链 A→B→C，不省略关键步骤
4. **关键机制怎么运作？** 不只命名，要说清内部逻辑（为什么 A 导致 B）
5. **用什么证据支撑？** 每个案例说明它在论证什么，不裸列
6. **读完能解决什么问题？** 读者视角的实际收获
7. **边界在哪里？** 不适用的场景，作者未覆盖的盲区

#### 写作规则

- **每条信息只出现一次**：决定它属于哪里，放在那里，其他地方不重复
- **机制在现象之前**：先说清怎么运作，再描述产生什么结果
- **案例不裸列**：每个案例前先说"这个案例说明了什么"，案例是论点的佐证，不是独立条目
- **连接全书各部分**：若书有上下篇或多个模块，明确写出它们之间的因果关系，不能各自独立悬空
- **链接不替代内容**：`[[概念页]]` 是深入阅读的入口，但核心内容必须在本页说清，不能用"详见概念页"代替解释
- **次要细节可略，核心脉络不可遗漏**：宁可删掉边缘案例，不能跳过关键推导步骤

## 数据源处理

特定数据源的获取和转换流程通过 **Skills** 按需加载，不在本文件中展开：

| 数据源 | Skill | 触发方式 |
|--------|-------|----------|
| YouTube 视频 | `/youtube <url>` | 手动或遇到 YouTube URL 自动触发 |
| EPUB 电子书 | `/epub <path>` | 手动或遇到 .epub 文件自动触发 |

Skill 文件位于 `~/.claude/skills/`（全局，跨知识库共用），包含完整的工具链、命令和踩坑经验。

## 大文件 Ingest 策略（书籍/长文档）

转换后的 markdown 可能很大，需在读取前判断策略。**Read 工具单次上限约 80 行（~2600 tokens）**，但多次 Read 的结果可以全部积累在同一个 context 里，最后一次性综合。

**估算公式**：`token 估算 ≈ 文件字节数 ÷ 4`（中英混合 markdown 经验值）

```bash
wc -c raw/书名_converted.md   # 查文件字节数，除以 4 得 token 估算
```

**架构原则**：主 Agent 只做 wiki 操作（读 index、读已有页面、写输出），不读入源文档正文。源文档交给子 Agent 读取并返回摘要。

### 策略一：主 Agent 直接读（源文档估算 < 50k tokens）

适用于文章、字幕、短报告等短文档：

1. `grep -n "^#" 文件` 确认结构（可选）
2. 并行发出所有 Read 调用，将全文载入 context
3. 综合写 wiki 页面

### 策略二：子 Agent 读取（源文档估算 ≥ 50k tokens）

适用于书籍、长报告等长文档：

1. `grep -n "^#" 文件` 扫描语义边界，按章节分组
2. **每组书籍内容 ≤ 150k tokens（约 600k 字节）**；保证整章不拆断，在此前提下尽量少分组
3. 无标题结构时（如纯文本），改用日期/序号等显式边界分组
4. 每组一个子 Agent，**组内并行发出所有 Read**，读完后**只输出本组摘要**（3-5k tokens），不写 wiki
5. 主 Agent 读取所有组的摘要，再读相关 wiki 页面，最后统一写读书笔记


## Ingest 操作规范

当收到新文档（文章、笔记、书摘、网页等）时，执行：

1. **读取 wiki/index.md**，了解当前已有哪些页面
2. **分析文档**，识别涉及的实体、概念、事件
3. **筛选关键图片**（如文档含图片引用）：
   - 扫描源 markdown 中的所有 `![](...)` 引用
   - **查看每张图片**，分为三类：
     - **关键图表**：数据表格、流程图、模型图、架构图、趋势曲线等——传递核心信息，文字难以替代
     - **辅助插图**：示意图、照片等——有参考价值但非必需
     - **装饰性图片**：封面、分隔符、出版信息、logo——跳过不引用
   - 记录关键图表的路径和所属上下文（在哪个章节/段落旁），后续步骤中写入 wiki 页面
4. **读取相关已有页面**（最多 3-5 个）
5. **更新或新建页面**：
   - 新内容：按模板新建页面，归入对应子目录
   - **已有页面必须实质更新，不能只加链接**：
     - 新文档提供了新的事实、数据、观点、指标 → 写入对应章节（核心内容/应用场景/确认指标等），标注来源和日期
     - 新文档提供了新的关联关系但无增量内容 → 仅更新关联页面章节
     - 判断标准：如果读者只看这个已有页面，能否感知到新文档带来的信息增量？如果不能，说明更新不到位
   - **嵌入关键图片**：在相关段落处插入 `![简要描述](raw/assets/书名/ImageXXX.jpg)`，并附一行文字说明图片内容
6. **更新 wiki/index.md**：新页面加入索引，更新摘要和时间
7. **追加 wiki/log.md**：`## [YYYY-MM-DD] ingest | 文档标题`
8. **移动原始文档**：将 inbox 中的原始文档（epub/网页/html 等）以及 pandoc 转换后的 `.md` 文件全部移至 `raw/`，确保 inbox 清空
9. **更新 qmd 索引**：运行 `qmd embed` 为新增/修改的页面生成向量索引
10. **Ingest 质量评分**（书籍/长篇内容时执行，短文章可跳过）：
    对本次读书笔记从三个维度各打 1-10 分，并追加到 `wiki/ingest_scores.md`：
    - **覆盖度**：核心问题、主张、框架、脉络、洞见是否有实质内容，有无明显遗漏
    - **简洁度**：有无冗余重复，表达是否精炼
    - **逻辑性**：论点是否连贯，框架是否清晰，读完能否理解作者推导

    `wiki/ingest_scores.md` 格式（追加一行）：
    ```
    | YYYY-MM-DD | 模型名 | 书名 | 覆盖度 | 简洁度 | 逻辑性 | 总分 | 备注 |
    ```
    评分后直接告知用户三项得分及简短理由。

11. **输出**：列出所有新建/修改的文件，标注引用了哪些图片

**注意：**
- 观点矛盾时，并列记录并注明来源，不擅自判断
- 数据信息标注来源和日期，避免过时数据覆盖新数据
- 一次 ingest 通常涉及 2-8 个页面
- 图片引用本地路径（`raw/assets/书名/filename.jpg`），避免依赖外部 URL
- 图片数量多时（>20张），优先筛选关键图表，辅助插图酌情引用，不必全部查看
- **ingest 完成后 inbox 中对应文档必须移走，不得残留**

## Query 操作规范

1. 读取 wiki/index.md 找相关页面
2. 读取相关页面内容
3. 基于 wiki 内容回答，标注引用（[[页面名]]）
4. **有价值的分析结论默认建议写回 wiki**（对比、洞察、发现的关联等），不应让知识停留在聊天记录里；由用户决定是否执行
5. 追加 log.md：`## [YYYY-MM-DD] query | 问题摘要`

## Lint 操作规范（每月或按需执行）

检查并报告：
- 孤立页面（无 inbound links）
- 数据超过 90 天未更新
- 页面间存在明显矛盾的观点
- 重要概念被提及但无独立页面
- 建议新增的对比页或综合页

追加 log.md：`## [YYYY-MM-DD] lint | 发现 N 个问题`

> log.md 格式说明：每条记录以 `## [YYYY-MM-DD]` 开头，便于用 `grep "^## \[" log.md` 解析历史。

## 搜索策略

- **小规模（< 100 页）**：直接读 index.md 导航，无需额外工具
- **中大规模（>= 100 页）**：考虑引入 [qmd](https://github.com/tobi/qmd)，提供本地 BM25/向量混合搜索，支持 CLI 和 MCP 两种接入方式

## 命名规范

- 人物：中文姓名或英文全名，如 `爱因斯坦.md`、`Alan_Turing.md`
- 公司/产品：常用名，如 `OpenAI.md`、`iPhone.md`
- 概念：准确术语，如 `熵.md`、`贝叶斯推断.md`
- 事件：简明描述，如 `2008年金融危机.md`、`阿波罗11号登月.md`
- 对比：`A_vs_B.md`
- 综合/宏观：中文名，如 `人工智能概览.md`

## 语言规范

- 默认用中文写 wiki 页面
- 专有名词、术语首次出现时附英文原文，如 `熵（Entropy）`
- 数字和数据标注单位和来源
