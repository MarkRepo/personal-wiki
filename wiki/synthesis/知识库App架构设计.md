## 核心结论

将个人 wiki 工作流产品化为面向非技术用户的知识库 App，核心设计决策是：**LLM 必须在服务端，用户完全无感**。App 把 LLM 作为基础设施内化，就像 Notion 用户不需要知道背后有数据库一样。商业模式为订阅制，LLM 推理成本通过月费消化。

## 详细分析

### 整体架构：Thin Client + Smart Backend

```
┌─────────────────────────────────────────────┐
│  Client (App / Web)                         │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐ │
│  │ Ingest  │  │ Wiki     │  │ Search &   │ │
│  │ Drop Zone│  │ Reader   │  │ Graph View │ │
│  └────┬────┘  └────┬─────┘  └─────┬──────┘ │
│       │            │               │        │
│  Local-first storage (SQLite/文件)          │
└───────┼────────────┼───────────────┼────────┘
        │ sync       │ read         │ query
┌───────┼────────────┼───────────────┼────────┐
│  Backend                                    │
│  ┌────┴─────┐  ┌───┴──────┐  ┌────┴──────┐ │
│  │ Ingest   │  │ Wiki     │  │ Semantic  │ │
│  │ Pipeline │  │ Storage  │  │ Index     │ │
│  └────┬─────┘  └──────────┘  └───────────┘ │
│       │                                     │
│  ┌────┴──────────────────┐                  │
│  │ LLM Orchestration     │                  │
│  │ (用户完全无感)          │                  │
│  └───────────────────────┘                  │
└─────────────────────────────────────────────┘
```

### LLM 部署位置分析

| 方案 | 优势 | 劣势 | 适合用户 |
|------|------|------|----------|
| **服务端 LLM（推荐）** | 零配置、质量稳定、持续迭代 | 需承担推理成本、隐私顾虑 | 非技术用户 |
| 客户端本地模型 | 隐私好、无持续成本 | 硬件要求高、质量差 | 极少数技术极客 |
| Bring Your Own Key | 不承担成本 | 用户需理解 API key 概念 | 开发者群体 |

**结论：目标用户连 API key 都搞不定，服务端 LLM 是唯一选择。**

### Ingest 管线设计——产品核心壁垒

#### 输入端：让「喂料」足够简单

| 入口 | 交互方式 |
|------|---------|
| 文件拖入 | epub、pdf、docx、图片 → 后端格式转换 |
| 链接粘贴 | 网页 URL、YouTube → 后端抓取/转录 |
| 剪藏插件 | 浏览器 extension，一键发送当前页面 |
| 手机分享 | 微信/系统分享菜单 → 转发到 App |
| 手动笔记 | App 内直接写，标记"需要整理" |

用户只需要做一个动作——**把东西扔进来**。

#### 处理端：LLM Pipeline（用户不可见）

```
原始内容
  ↓
[Stage 1] 格式标准化（pandoc/trafilatura/whisper）
  ↓
[Stage 2] LLM 分析：识别实体、概念、事件、观点
  ↓
[Stage 3] LLM 与现有知识库对比：
          - 哪些页面需要更新？
          - 哪些是新知识点？
          - 有无矛盾观点？
  ↓
[Stage 4] 生成 diff（不直接写入）
  ↓
[Stage 5] 更新索引和关联
```

**关键设计：Stage 4 输出 diff，用户审阅后确认。**

用户看到的是类似 PR review 的界面：

```
📥 Ingest: 《纳瓦尔宝典》第3章

新建页面：
  + concepts/杠杆效应.md
  + concepts/四种运气.md

更新页面：
  ~ entities/纳瓦尔·拉维坎特.md   (+3 段落)
  ~ synthesis/纳瓦尔宝典读书笔记.md (+1 章节)

[全部接受]  [逐条审阅]  [丢弃]
```

这给了用户**掌控感**，同时不要求任何技术能力。想仔细看就逐条审，信任系统就一键接受。

#### Schema 层：可选模板 + 自动推断

- 提供预设 schema：读书笔记型、学术研究型、行业观察型、个人成长型
- 允许自定义分类，但提供合理默认值
- LLM 自动判断归类，用户只在不满意时调整
- 大多数用户不会设计 schema，默认模板要足够好

### 阅读端：Local-first

Wiki 内容同步到本地，离线可读：

- **移动端**：类似 Obsidian Mobile，本地 markdown 渲染
- **Web 端**：类似 Quartz，数据从云端同步
- **搜索**：本地全文搜索 + 云端语义搜索（向量检索）
- **图谱**：关联可视化，知识库相比笔记工具的差异化功能

### 成本模型

- 一次 ingest ≈ 10-30K tokens input + 5-10K output
- 按 Claude Sonnet 价格约 $0.01-0.05/次
- 普通用户每月 ingest 50-200 次
- **每用户每月 LLM 成本 ≈ $1-5**
- 订阅价 $10-15/月可覆盖，免费版限制每月 20 次 ingest

### 最难的两个问题

1. **Ingest 质量**：当前 wiki 的 CLAUDE.md 规范经过大量调试才稳定产出。产品化时这套 prompt engineering 需要对各类输入都稳定输出高质量结果——这是核心壁垒，也是需要持续迭代的部分。

2. **冷启动体验**：空知识库没有吸引力。需要让用户在 ingest 3-5 篇内容后就能感受到「知识之间产生了连接」的价值。可能需要在早期主动推荐关联、生成初始综合页。
